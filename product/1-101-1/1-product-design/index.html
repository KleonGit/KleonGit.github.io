产品设计文档

Keypoint:

  • 统一的信息，数据，IO管线

领域模型

User

基本操作单位，通常由自然人担当，也可以是消息源。

People

自然人注册用户。

Bot [Pipe]

具有原子化处理能力的自动化插件。

具有特定功能的消息处理程序，可以在Bot Factory中定义Bot Pipeline行为。

Bot可由Blueprint实例化一个具体行为和参数的实体，挂载在Channel上。

Channel中Bot的预设包括：

  • 对所有消息处理
  • @Bot 处理
  • 关键词匹配处理
  • 时间触发
  • 定位触发
  • 定义命令

Bot的基本组成：

  • 触发器
    • 定时触发
    • webhook
      • type
      • content
      • namepsace
    • 内部调用
  • Plugin-A
  • Plugin-B
  • 推送Block/Message/Row
    • 绑定namespace
    • 动态namespace

Bot的设计称为Blueprint，实例化称为Bot，BotAccount为一种User。

Role

权限管理。参考kubernetes。

Exchange权限，Doc权限，Bot权限。

以/team/node/node为权限基本单元。

Block [Drop]

Block(Water Drop)是H2O基本单元，其生命周期包括输入，Channel消息以及文档，并与数据库的行关联。

Block的最基础形式是一行文本，可以接受markdown语法，或限定的与markdown对应的富文本语法。

Block支持多种原生组件：

  • [x] Markdown
  • 图片
  • 声音
  • 表格
  • 表格卡片
  • 图表
  • 表单卡片(?)
  • Third Party Link
    • Table
    • Doc (Form 匿名？)
  • 第三方Block?动态图表，主题，个性化。安全性x。

Message

Channel中Block具有Message的性质，包括永久Message，临时Message。

永久Message在Channel中所有用户可见，包括用户间对话，用户与ActionBot的指令等。
临时Message多为系统提示，不会在数据库中保存。

Module

Exchange

Exchange提供消息的收集，预处理，中转，分发，输出功能。

使用IM模型作为Exchange功能参考。

Exchange是消息集散中心，用户的命令，功能的轻UI都可以都过Exchange完成，并留档。

Exchange界面可以作为简易的命令行，交互底层基于Bot(第三方软件提供服务，Action，Pipeline)，Table(数据空间)，Doc(书面展示与交互)。

Team

Exchange的最顶层组织结构。

Team内的成员共享权限管理。

个人默认创建并关联一个Team，不可删除。

Channel

基本的会话节点。

消息的输入管线。

协作的基本空间。

Thread

细化的会话组织结构。方便全选。

Doc

基本的展示节点。

Table

基本的数据节点。

/

基本的空节点，用于组织目录结构。

Doc

Message可直接和文档连接。

Message中的消息可直接和文档中的block同步。复制？引用？

Block与Message的Block概念同步。

Doc可以看成Table的一个附有说明的状态截面。

Document

静态载体

Slice

Slice为Doc的一种播放形式。

Form

可交互载体 关联Action

数据库

保存有序序列

  • 增加order字段，每次增删查改都批量修改序号。
  • 使用string代替order，比如’a’，’aa’。每次向后append。
  • 使用gap代替order，比如100,200。每次取中间数。
  • 使用链表法。
